## Original Instructions

# Claude Code 플러그인을 위한 Agent Development

## Overview

Agent는 복잡하고 여러 단계로 구성된 작업을 독립적으로 처리하는 자율적인 하위 프로세스(autonomous subprocess)입니다. 에이전트의 구조, 트리거 조건 및 system prompt 설계를 이해하면 강력한 자율 기능을 만들 수 있습니다.

**핵심 개념:**
- Agent는 자율적인 작업을 위한(FOR) 것이고, command는 사용자가 시작하는 작업을 위한(FOR) 것입니다.
- YAML frontmatter가 포함된 마크다운 파일 형식입니다.
- 예시가 포함된 description 필드를 통해 트리거됩니다.
- System prompt가 에이전트의 동작을 정의합니다.
- Model 및 color를 커스터마이징할 수 있습니다.

## Agent File Structure

### Complete Format

```markdown
---
name: agent-identifier
description: Use this agent when [triggering conditions]. Examples:

<example>
Context: [Situation description]
user: "[User request]"
assistant: "[How assistant should respond and use this agent]"
<commentary>
[Why this agent should be triggered]
</commentary>
</example>

<example>
[Additional example...]
</example>

model: inherit
color: blue
tools: ["Read", "Write", "Grep"]
---

You are [agent role description]...

**Your Core Responsibilities:**
1. [Responsibility 1]
2. [Responsibility 2]

**Analysis Process:**
[Step-by-step workflow]

**Output Format:**
[What to return]
```

## Frontmatter Fields

### name (required)

네임스페이스 지정 및 호출에 사용되는 에이전트의 identifier입니다.

**포맷:** 소문자, 숫자, 하이픈만 허용
**길이:** 3-50자
**패턴:** 영숫자(alphanumeric)로 시작하고 끝나야 함

**올바른 예시:**
- `code-reviewer`
- `test-generator`
- `api-docs-writer`
- `security-analyzer`

**잘못된 예시:**
- `helper` (너무 일반적임)
- `-agent-` (하이픈으로 시작하거나 끝남)
- `my_agent` (밑줄은 허용되지 않음)
- `ag` (너무 짧음, 3자 미만)

### description (required)

Claude가 언제 이 에이전트를 트리거해야 하는지 정의합니다. **가장 중요한 필드입니다.**

**필수 포함 사항:**
1. 트리거 조건 ("Use this agent when...")
2. 사용법을 보여주는 여러 개의 `<example>` blocks

3. 각 예시의 Context, 사용자 요청(user), 어시스턴트 응답(assistant)
4. 에이전트가 트리거되는 이유를 설명하는 `<commentary>`

**포맷:**
```
Use this agent when [conditions]. Examples:

<example>
Context: [Scenario description]
user: "[What user says]"
assistant: "[How Claude should respond]"
<commentary>
[Why this agent is appropriate]
</commentary>
</example>

[More examples...]
```

**베스트 프랙티스:**
- 2~4개의 구체적인 예시를 포함하십시오.
- 프로액티브(proactive) 및 리액티브(reactive) 트리거를 모두 보여주십시오.
- 동일한 의도의 다양한 표현 방식을 다루십시오.
- commentary에서 이유를 설명하십시오.
- 에이전트를 사용하지 말아야 할 때(NOT to use)를 명시하십시오.

### model (required)

에이전트가 사용할 모델을 지정합니다.

**옵션:**
- `inherit` - 상위(parent)와 동일한 모델 사용 (권장)
- `sonnet` - Claude Sonnet (균형 잡힌 성능)
- `opus` - Claude Opus (가장 강력함, 비용 높음)
- `haiku` - Claude Haiku (빠르고 저렴함)

**권장 사항:** 에이전트에 특정 모델 기능이 필요하지 않은 한 `inherit`을 사용하십시오.

### color (required)

UI에서 에이전트를 시각적으로 구별하기 위한 식별자입니다.

**옵션:** `blue`, `cyan`, `green`, `yellow`, `magenta`, `red`

**가이드라인:**
- 동일한 플러그인 내에서 다른 에이전트들과 겹치지 않게 고유한 색상을 선택하십시오.
- 유사한 유형의 에이전트에는 일관된 색상을 사용하십시오.
- Blue/cyan: 분석(Analysis), 검토(review)
- Green: 성공 지향 작업
- Yellow: 경고, 검증(validation)
- Red: 중요(Critical), 보안(security)
- Magenta: 크리에이티브, 생성(generation)

### tools (optional)

에이전트가 사용할 수 있는 도구를 제한합니다.

**포맷:** 도구 이름의 배열

```yaml
tools: ["Read", "Write", "Grep", "Bash"]
```

**기본값:** 생략하면 에이전트가 모든 도구에 액세스할 수 있습니다.

**베스트 프랙티스:** 필요한 최소한의 도구로 제한하십시오 (최소 권한 원칙).

**일반적인 도구 세트:**
- 읽기 전용 분석: `["Read", "Grep", "Glob"]`
- 코드 생성: `["Read", "Write", "Grep"]`
- 테스트: `["Read", "Bash", "Grep"]`
- 모든 권한: 필드를 생략하거나 `["*"]` 사용

## System Prompt Design

마크다운 본문은 에이전트의 system prompt가 됩니다. 에이전트에게 직접 말하듯이 2인칭으로 작성하십시오.

### Structure

**표준 템플릿:**
```markdown
You are [role] specializing in [domain].

**Your Core Responsibilities:**
1. [Primary responsibility]
2. [Secondary responsibility]
3. [Additional responsibilities...]

**Analysis Process:**
1. [Step one]
2. [Step two]
3. [Step three]
[...]

**Quality Standards:**
- [Standard 1]
- [Standard 2]

**Output Format:**
Provide results in this format:
- [What to include]
- [How to structure]

**Edge Cases:**
Handle these situations:
- [Edge case 1]: [How to handle]
- [Edge case 2]: [How to handle]
```

### Best Practices

✅ **권장 사항 (DO):**
- 2인칭("You are...", "You will...")으로 작성하십시오.
- 책임 사항을 구체적으로 기재하십시오.
- 단계별 프로세스를 제공하십시오.
- 출력 형식을 정의하십시오.
- 품질 기준을 포함하십시오.
- 예외 상황(edge cases)을 처리하십시오.
- 10,000자 미만으로 유지하십시오.

❌ **금지 사항 (DON'T):**
- 1인칭("I am...", "I will...")으로 작성하지 마십시오.
- 모호하거나 일반적인 표현은 피하십시오.
- 프로세스 단계를 누락하지 마십시오.
- 출력 형식을 정의하지 않은 채로 두지 마십시오.
- 품질 가이드를 생략하지 마십시오.
- 에러 케이스를 무시하지 마십시오.

## Creating Agents

### Method 1: AI-Assisted Generation

이 프롬프트 패턴을 사용하십시오 (Claude Code에서 추출됨):

```
Create an agent configuration based on this request: "[YOUR DESCRIPTION]"

Requirements:
1. Extract core intent and responsibilities
2. Design expert persona for the domain
3. Create comprehensive system prompt with:
   - Clear behavioral boundaries
   - Specific methodologies
   - Edge case handling
   - Output format
4. Create identifier (lowercase, hyphens, 3-50 chars)
5. Write description with triggering conditions
6. Include 2-3 <example> blocks showing when to use

Return JSON with:
{
  "identifier": "agent-name",
  "whenToUse": "Use this agent when... Examples: <example>...</example>",
  "systemPrompt": "You are..."
}
```

그런 다음 frontmatter가 포함된 에이전트 파일 형식으로 변환합니다.

전체 템플릿은 `examples/agent-creation-prompt.md`를 참고하십시오.

### Method 2: Manual Creation

1. 에이전트 identifier를 선택합니다 (3-50자, 소문자, 하이픈).
2. 예시와 함께 description을 작성합니다.
3. 모델을 선택합니다 (일반적으로 `inherit`).
4. 시각적 구분을 위한 색상을 선택합니다.
5. 도구를 정의합니다 (액세스를 제한하려는 경우).
6. 위의 구조에 따라 system prompt를 작성합니다.
7. `agents/agent-name.md`로 저장합니다.

## Validation Rules

### Identifier Validation

```
✅ Valid: code-reviewer, test-gen, api-analyzer-v2
❌ Invalid: ag (too short), -start (starts with hyphen), my_agent (underscore)
```

**규칙:**
- 3~50자
- 소문자, 숫자, 하이픈만 허용
- 반드시 영숫자(alphanumeric)로 시작하고 끝나야 함
- 밑줄, 공백, 특수 문자 금지

### Description Validation

**길이:** 10~5,000자
**필수 포함:** 트리거 조건 및 예시
**권장:** 2~4개 예시를 포함하여 200~1,000자

### System Prompt Validation

**길이:** 20~10,000자
**권장:** 500~3,000자
**구조:** 명확한 책임, 프로세스, 출력 형식

- 하위 디렉토리가 있는 경우: `plugin:subdir:agent-name`

## Testing Agents

### Test Triggering

에이전트가 올바르게 트리거되는지 확인하기 위한 테스트 시나리오를 만듭니다:

1. 특정 트리거 예시가 포함된 에이전트를 작성합니다.
2. 테스트에서 예시와 유사한 문구를 사용합니다.
3. Claude가 에이전트를 로드하는지 확인합니다.
4. 에이전트가 예상된 기능을 제공하는지 검증합니다.

### Test System Prompt

system prompt가 완전한지 확인합니다:

1. 에이전트에게 일반적인 작업을 부여합니다.
2. 프로세스 단계를 따르는지 확인합니다.
3. 출력 형식이 올바른지 검증합니다.
4. 프롬프트에 언급된 예외 상황(edge cases)을 테스트합니다.
5. 품질 표준이 충족되는지 확인합니다.

### Best Practices

**권장 사항 (DO):**
- ✅ description에 2~4개의 구체적인 예시를 포함하십시오.
- ✅ 명확한 트리거 조건을 작성하십시오.
- ✅ 특별한 요구 사항이 없으면 모델에 `inherit`을 사용하십시오.
- ✅ 적절한 도구를 선택하십시오 (최소 권한 원칙).
- ✅ 구조화되고 명확한 system prompt를 작성하십시오.
- ✅ 에이전트 트리거를 철저히 테스트하십시오.

**금지 사항 (DON'T):**
- ❌ 예시 없이 일반적인 description을 사용하지 마십시오.
- ❌ 트리거 조건을 누락하지 마십시오.
- ❌ 모든 에이전트에 동일한 색상을 지정하지 마십시오.
- ❌ 불필요한 도구 권한을 부여하지 마십시오.
- ❌ 모호한 system prompt를 작성하지 마십시오.
- ❌ 테스트를 건너뛰지 마십시오.

## Implementation Workflow

플러그인용 에이전트를 만들려면:

1. 에이전트 목적 및 트리거 조건을 정의합니다.
2. 생성 방법(AI 지원 또는 수동)을 선택합니다.
3. `agents/agent-name.md` 파일을 생성합니다.
4. 모든 필수 필드가 포함된 frontmatter를 작성합니다.
5. 베스트 프랙티스에 따라 system prompt를 작성합니다.
6. description에 2~4개의 트리거 예시를 포함합니다.
7. `scripts/validate-agent.sh`로 유효성을 검사합니다.
8. 실제 시나리오로 트리거를 테스트합니다.
9. 플러그인 README에 에이전트를 문서화합니다.

자율적인 작동을 위해 명확한 트리거 조건과 포괄적인 system prompt 작성에 집중하십시오.
